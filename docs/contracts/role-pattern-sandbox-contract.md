## 1. Layered enforcement model (Layer 2 / 3 / 4)

The sandbox filesystem boundary for a role × pattern is enforced by **three
distinct layers**, each with different scope, bypass conditions, and tool
applicability. **Layer 1 (Claude Code core safety classifier) is named for
completeness but is not itself a contract surface here** — Layer 1 is the
upstream classifier that ships with Claude Code and is not configurable per role.

| Layer | Mechanism | Where configured | Tools controlled | Bypass conditions |
|---|---|---|---|---|
| **Layer 2** | `permissions.allow` / `permissions.deny` | `.claude/settings.local.json` (per-role) — generated by `claude-org-runtime settings generate` from `tools/org_extension_schema.json` `worker_roles[*]` (concrete templates) and `roles[*]` (audit constraints) | All Claude Code tool invocations: `Read`, `Edit`, `Write`, `MultiEdit`, `NotebookEdit`, `Bash`, `mcp__*`, etc. | **`permission_mode=bypassPermissions` disables Layer 2 entirely** (the dispatcher uses this mode by Sonnet/spawn-classifier necessity; for that role Layer 2 is a no-op and Layer 4 is the only enforcement, see §3.2). |
| **Layer 3** | `sandbox.filesystem.additionalDirectories` / `denyRead` / `denyWrite` (bubblewrap mount-namespace) | Same `.claude/settings.local.json` `sandbox` block — **Phase 1 (#378) target only**; not yet emitted by `claude-org-runtime settings generate` (a `sandbox` field in the bundled schema is still pending). Today the sandbox block is **handcraft** in [`docs/sandbox-probe/profiles/profile-tightened.json`](../sandbox-probe/profiles/profile-tightened.json) and `profile-baseline.json`. | All filesystem syscalls inside the sandboxed Claude process — including `Bash` redirections (`>`, `>>`, `tee`), tool-generated temp files, child processes, and any path resolution Claude makes. | **Adaptive suppression on symlink-escape platforms** (runtime 0.1.4 `render_role_with_metadata`): on WSL where `~/.aws` / `~/.ssh` resolve to `/mnt/c/...` outside the sandbox view, the suppressed deny entries are emitted only as `permissions.deny` — see §1.3 caveat. `dangerouslyDisableSandbox: true` and `failIfUnavailable: false` + bwrap absence both fall-open Layer 3. |
| **Layer 4** | `PreToolUse` hook scripts at `.hooks/*.sh` invoked via `hooks.PreToolUse` blocks in `.claude/settings.local.json` | Same `.claude/settings.local.json` `hooks` block — emitted by the generator as part of each `worker_roles[*]` / role template. Hooks themselves live in `.hooks/` in the repo. | Hooks are scoped per-`matcher`: `check-worker-boundary.sh` and `block-org-structure.sh` (Edit/Write portion) and `block-dispatcher-out-of-scope.sh` are **`Edit \| Write` (`NotebookEdit` for the dispatcher hook) matchers** — they receive `tool_input.file_path` and inspect it. `block-git-push.sh`, `block-dangerous-git.sh`, `block-no-verify.sh` and the Bash portion of `block-org-structure.sh` are **`Bash` matcher** — they receive `tool_input.command` (a shell string) and inspect it. | A hook can only block the tool invocation it is matched against. A hook cannot retroactively roll back a write that has already happened on a different tool path. **Layer 4 stays active under `bypassPermissions`** — that is exactly why it is the dispatcher's only enforcement layer. |

### 1.1 Why three layers, not two

Path-level `permissions.deny` (Layer 2) and `sandbox.filesystem.denyRead/denyWrite`
(Layer 3) **deny different things even when their pattern strings look identical**:

- Layer 2 `Read(~/.aws/*)` denies `Read` *tool* calls whose argument matches the
  pattern. `Bash(cat ~/.aws/credentials)` is **not blocked by Layer 2's `Read()`
  rule** — it is a `Bash` tool call whose internal `cat` syscall reads the file.
  To block that path Layer 2 would need a `Bash(cat ~/.aws/*)` deny, which is
  brittle (any of `head`, `less`, `xxd`, redirect, `python -c "open(...)"`,
  etc. would bypass).
- Layer 3 `denyRead` is enforced at the bubblewrap mount-namespace level, which
  catches **any syscall** the sandboxed process makes — `Bash(cat ...)`,
  tool-generated child process reads, redirections, etc. This is why
  credential paths *should* be Layer 3 for defense-in-depth even when Layer 2
  also names them.
- Layer 4 `PreToolUse` hooks see only the tool invocation they are matched
  against. `check-worker-boundary.sh` enforces an `Edit|Write` allow-list
  (`tool_input.file_path` must be inside `WORKER_DIR` or a small carve-out
  set). It does **not** see `Bash` writes; that is why
  `block-org-structure.sh` is wired to **both** matchers separately (the
  Edit/Write half and the Bash half are duplicate-installed against the same
  script with branch-on-`tool_name`).

Phase 1 #378 will mechanize Layer 3 (today's gap layer); Phase 0 fixes the
prescriptive surface so the schema PR knows what to emit.

### 1.2 Tool-attribution table for hooks (Codex review #377 Nit 1)

| Hook script | PreToolUse matcher | Tool input field examined | Decision rule |
|---|---|---|---|
| `.hooks/check-worker-boundary.sh` | `Edit \| Write` (some role configs add `MultiEdit`) | `tool_input.file_path` | Allow if `realpath(file_path)` is inside `WORKER_DIR/`, equals `~/.claude/plans/<...>`, or is `CLAUDE_ORG_PATH/knowledge/raw/YYYY-MM-DD-<topic>.md` (kebab-case). Else exit 2. |
| `.hooks/block-org-structure.sh` (Edit/Write half) | `Edit \| Write` | `tool_input.file_path` | Deny if `file_path` is `<WORKER_DIR>/.claude\|.dispatcher\|.foreman\|.curator\|.state\|registry\|dashboard\|knowledge` (top-level only for the second three). Carve-outs: `CLAUDE_ORG_PATH/knowledge/raw/YYYY-MM-DD-*.md` and `WORKER_DIR/.claude/plans/`. |
| `.hooks/block-org-structure.sh` (Bash half) | `Bash` | `tool_input.command` (shell string, slash-normalized) | Deny if command string contains `mkdir|touch|cp|mv` followed by one of the org-structure dirnames; same `knowledge/raw/` and `.claude/plans/` carve-outs. |
| `.hooks/block-dispatcher-out-of-scope.sh` | `Edit \| Write \| NotebookEdit` | `tool_input.file_path` | Allow only `<CLAUDE_ORG_PATH>/.dispatcher/`, `<CLAUDE_ORG_PATH>/.state/`, and `<CLAUDE_ORG_PATH>/knowledge/raw/YYYY-MM-DD-<topic>.md`. Else exit 2. |
| `.hooks/block-git-push.sh` | `Bash` | `tool_input.command` | Deny if the command string contains a `git ... push ...` invocation (also `eval "..."` / `bash -c "..."` unwrapping). |
| `.hooks/block-dangerous-git.sh` | `Bash` | `tool_input.command` | Per-segment deny: `git push --force / -f / --force-with-lease`, `git reset --hard`, `git branch -D / --delete --force`. Segment splitter respects quoted separators. |
| `.hooks/block-no-verify.sh` | `Bash` | `tool_input.command` | Deny `git commit / push --no-verify` (incl. `eval` / `bash -c` unwrapping and known-`VAR=value` expansion). |
| `.hooks/block-workers-delete.sh` | `Bash` | `tool_input.command` | Deny `rm -rf` (and equivalents) targeted at `<workers_dir>/...`. Used by the secretary role; not enumerated row-by-row in this contract because its surface is symmetric across patterns. |

The script names + matcher pairing above is the source-of-truth for "which layer
enforces what" in the per-role tables that follow. **Path-only and
command-only enforcement are not mixed in the same row** of those tables.

### 1.3 WSL / symlink-escape adaptive Layer 3 suppression (Codex Major 4)

`claude-org-runtime` 0.1.4 (`claude-org-runtime#10`, ja PR #397) introduced
`render_role_with_metadata` which detects when a Layer 3 path entry resolves to
a symlink that escapes the sandbox view (typically `~/.aws → /mnt/c/Users/...`
on WSL2). For such entries it:

1. **Suppresses** the Layer 3 `denyRead` / `denyWrite` entry (the bwrap mount
   would otherwise fail with `Can't mount tmpfs on /newroot/home/<user>/.aws:
   No such file or directory` — observed in
   [`docs/sandbox-probe/notes/iteration-b-round3-results.md`](../sandbox-probe/notes/iteration-b-round3-results.md)
   §4.3).
2. **Always emits** the corresponding `permissions.deny` Layer 2 entry
   (`Read(~/.aws/*)` / `Read(~/.ssh/*)`) so secret reads via the `Read` tool
   stay blocked at Layer 2.
3. Annotates the emitted profile's `$comment` with `platform=wsl, layer-3
   entries suppressed: [...]` and emits a `sandbox_deny_skipped` journal event
   (Phase 3 case E specification, [`docs/sandbox-probe/phase3-bootstrap-policy-design.md`](../sandbox-probe/phase3-bootstrap-policy-design.md)
   §5.2).

**Phase 0 contract treatment**: rows that name `~/.aws/**`, `~/.ssh/**`,
`~/.claude/**` (or any path that resolves outside the sandbox view) under
Layer 3 are written as **conditional** rather than absolute. The row reads:

> Layer 3 `denyRead`: `~/.aws/**` (suppressed-on-symlink-escape; falls back to
> Layer 2 `Read(~/.aws/*)`).

A Phase 0 reader must not interpret a Layer 3 `~/.aws/**` cell as
"unconditionally denied at Layer 3" — that contradicts the released runtime
behavior on WSL. Phase 1 mechanization will encode the same conditional in the
schema (likely as a `platform_conditional` flag on the `sandbox.denyRead`
entry).

### 1.4 What this contract does NOT enumerate

This contract is **filesystem read/write boundary** scoped. It does **not**
enumerate:

- **Process / network / mcp surfaces** — those are governed by
  `permissions.allow` for `Bash(...)`, `mcp__*`, etc., and are out of Phase 0
  scope. They are tracked separately under the role-contract Set A
  (`docs/contracts/role-contract.md`) "Constraints" sections.
- **`required_hook_scripts`-level invariants** that apply uniformly to every
  role — those live in `tools/org_extension_schema.json` top-level
  `required_hook_scripts` and are validated by `tools/check_role_configs.py`.
  This contract assumes those invariants hold.
- **`.state/journal.jsonl` schema** — see
  `docs/contracts/state-schema-contract.md` and `docs/journal-events.md`.
- **Per-role MCP tool allowlists** — those are filesystem-adjacent (some MCP
  tools write to disk) but Phase 0 keeps them out of the row tables to avoid
  duplicating Set A's `permissions.allow` enumeration.

## 2. Pattern reference (recap)

These are the directory patterns the secretary's `tools/gen_delegate_payload.py`
resolver chooses between (per
[`.claude/skills/org-delegate/references/delegate-flow-details.md`](../../.claude/skills/org-delegate/references/delegate-flow-details.md)
§1). Each row in §3 / §4 below is keyed by `(role, pattern, pattern_variant?)`.

| Pattern | `pattern_variant` | `worker_dir` | Brief filename | `.git/` topology |
|---|---|---|---|---|
| **A** — project dir | `null` | `<workers_dir>/<project_slug>/` | `CLAUDE.md` | Independent clone (or reuse) of project repo. Worker has its own `.git/` directory. |
| **B** — worktree (default) | `null` | `<workers_dir>/<project_slug>/.worktrees/<task_id>/` | `CLAUDE.md` | `.git` **file** pointing into the project clone's `.git/worktrees/<task_id>/`. Common object store at `<workers_dir>/<project_slug>/.git/objects/`. |
| **B** — `live_repo_worktree` (Issue #289) | `live_repo_worktree` | `<claude_org_path>/.worktrees/<task_id>/` | `CLAUDE.local.md` (per `claude-org-self-edit.md` §2) | `.git` **file** pointing into `<claude_org_path>/.git/worktrees/<task_id>/`. Single shared `.git/` between secretary and worker. **Self-edit role only.** |
| **C** — ephemeral | `null` | `<workers_dir>/<task_id>/` | `CLAUDE.md` | Worker `git init` inside `worker_dir`, or no git at all. No coupling to a project clone. |
| **C** — `gitignored_repo_root` (Step 0.7) | `gitignored_repo_root` | `<existing_repo_root>/` (registered project's repo root) | `CLAUDE.local.md` | Worker reuses the existing repo's `.git/`. **No new branch is created** (`planned_branch=null`). Same-repo concurrency is forbidden (file-name collision on `CLAUDE.local.md` / `.claude/settings.local.json`). |

Pattern A vs Pattern B is **not a role-controlled choice** — it falls out of the

`registry/projects.md`. The role × pattern combinations this contract enumerates
are therefore:

| Role | Patterns covered in §3 / §4 |
|---|---|
| Secretary | (no pattern dimension) |
| Dispatcher | (no pattern dimension) |
| Curator | (no pattern dimension) |
| Worker `default` | A, B (default), C, C-`gitignored_repo_root` |
| Worker `claude-org-self-edit` | B-`live_repo_worktree`, C-`gitignored_repo_root` (when self-edit target is gitignored) |
| Worker `doc-audit` | A or B or C — pattern is orthogonal; the doc-audit row is **role-only** because the audit constraint dominates |

## 3. Per-role contracts (no pattern dimension)

### 3.1 Secretary

The secretary runs in the live repo at `<claude_org_path>/`, in normal Claude
Code permission mode (per-tool prompts). It is the only role that can write to
`registry/`, full `.state/` (incl. `.state/workers/<task>.md` content beyond
the dispatcher's automated rows), and it is the role that places worker
briefs and `settings.local.json` files.

#### 3.1.1 Read surface (prescriptive)

- `<claude_org_path>/**` — full read access required for delegation, registry
  resolution, journal inspection, project audit.
- `<claude_org_path>/../workers/**` — read of any worker dir under
  `workers_dir` for status confirmation. (The path `../workers/` is the
  default `workers_dir` per `registry/org-config.md`; if relocated, the
  configured `workers_dir` is the prescriptive surface.)
- `~/.claude/plans/` — Claude Code internal plan files.

**Denied reads (prescriptive)**:

- `Read(.env)`, `Read(.env.*)`, `Read(**/credentials*)`, `Read(**/*.pem)`,
  `Read(~/.config/gh/hosts.yml)` — see §1.3 for the WSL Layer 3 caveat on
  `~/.aws` / `~/.ssh`.

#### 3.1.2 Write surface (prescriptive)

| Surface | Allowed by | Notes |
|---|---|---|
| `<claude_org_path>/.state/**` | Direct | Journal append must go through `tools/journal_append.sh\|py`; raw `>>` is an org-discipline ban (see role-contract Set A §secretary). |
| `<claude_org_path>/registry/**` | Direct | `registry/projects.md`, `registry/org-config.md`. |
| `<claude_org_path>/dashboard/**` | Direct | Snapshot regeneration via `dashboard/org_state_converter.py`. |
| `<claude_org_path>/knowledge/**` | Direct | Limited in practice to retro-related files and `knowledge/skill-candidates.md`; raw learnings are typically worker / dispatcher / curator owned. |
| `<workers_dir>/<task>/CLAUDE.md` and `<task>/.claude/settings.local.json` | Generator-only | Worker brief + settings file placement during `org-delegate` Step 1.5. **`Edit` / `Write` to `*/workers/*/.claude/settings.local.json` is denied at Layer 2** (`required_deny` in `org_extension_schema.json` `roles.secretary`); the canonical generator is `claude-org-runtime settings generate`. Manual JSON authoring is rejected by drift CI. |
| `<workers_dir>/<task>/.worktrees/.../.claude/settings.local.json` | Generator-only | Same constraint; required_deny rule covers the worktree path too. |
| `<claude_org_path>/.worktrees/<task>/.claude/settings.local.json` | Generator-only | Same constraint applies for the live_repo_worktree variant; the secretary's `roles.secretary.required_deny` rule is keyed on the file-path glob, so it extends to the live repo's worktree subtree as well. |

**Denied writes (prescriptive)**:

- `Write(*/workers/*/.claude/settings.local.json)` — secretary must call
  `claude-org-runtime settings generate` instead. Layer 2 `required_deny`
  enforces.
- `Edit(*/workers/*/.claude/settings.local.json)` — same as above.
- `Write(*/workers/*/.worktrees/*/.claude/settings.local.json)` and the `Edit`
  twin — same.
- `git push` (raw or via `eval`/`bash -c`) — Layer 4 `block-git-push.sh` is
  installed on the secretary as well? **No** — the secretary is the role that
  *does* push. The hook is installed only on dispatcher / worker. The
  secretary's `git push` discipline is enforced by user judgment + the
  `org-pull-request` skill flow (see role-contract.md §secretary). This row
  is named here only to document the boundary.

#### 3.1.3 Layer attribution (current state)

| Surface | Layer 2 | Layer 3 | Layer 4 |
|---|---|---|---|
| `*/workers/*/.claude/settings.local.json` write | `required_deny` ✔ | n/a (Phase 1 #378) | none |
| `<claude_org_path>/.state/**` write | allowed via `Bash(python:*)` etc. | n/a | `block-no-verify.sh` (Bash) covers `git commit --no-verify` only |
| `~/.aws/**` read | `Read(~/.aws/*)` ✔ (when added; not in default secretary template) | adaptive (see §1.3); not currently emitted for secretary | none |
| Org-structure dir creation under workers | n/a (secretary is *not* a worker) | n/a | `block-workers-delete.sh` (`Bash` matcher) for workers_dir delete only |

**Gap → Phase 1 #378**: Secretary does not currently have a `sandbox` block.
Phase 1 should emit one with `additionalDirectories: [<claude_org_path>]` and
`denyRead` covering credential paths (with §1.3 adaptive rules), so that
Bash-mediated credential reads from secretary code are also blocked. The
secretary today relies on Layer 2 + user judgment.

### 3.2 Dispatcher

The dispatcher runs at `<claude_org_path>/.dispatcher/` with `permission_mode=
bypassPermissions`. **Layer 2 is therefore a no-op for this role**. All
filesystem boundary enforcement is Layer 4.

#### 3.2.1 Read surface (prescriptive)

- `<claude_org_path>/.dispatcher/**` — its own home.
- `<claude_org_path>/.state/**` — read inbox / outbox / cursor / worker logs.
- `<claude_org_path>/knowledge/raw/` — needed by retro skill.
- `<claude_org_path>/registry/**` — needed for project lookup during
  `delegate-plan`.
- `<claude_org_path>/tools/**` — read helper sources.

#### 3.2.2 Write surface (prescriptive)

The dispatcher's write surface is **strictly enumerated** (Codex review Major 1
demands path-unit separation; the Layer 4 hook is the only enforcement):

| Surface | Allowed via | Layer |
|---|---|---|
| `<claude_org_path>/.dispatcher/**` | Edit/Write | Layer 4 (`block-dispatcher-out-of-scope.sh` allow rule 1) |
| `<claude_org_path>/.state/**` | Edit/Write | Layer 4 (`block-dispatcher-out-of-scope.sh` allow rule 2). Journal append must go through `tools/journal_append.sh\|py`. |
| `<claude_org_path>/knowledge/raw/YYYY-MM-DD-<kebab-topic>.md` | Edit/Write | Layer 4 (allow rule 3, kebab-case basename validated) |
| `git commit` in `.dispatcher/` | Bash | n/a — dispatcher does not run `git commit` on its own files in steady state; this row is reserved for future sub-skills |

**Denied writes**:

- `<claude_org_path>/tools/**`, `dashboard/**`, `tests/**`,
  `.claude/skills/**`, `docs/**`, `registry/**`, **anything under
  `<workers_dir>/**`** — Layer 4 `block-dispatcher-out-of-scope.sh` denies all
  `Edit | Write | NotebookEdit` outside the three allow rules above.
- `git push` — Layer 4 `block-git-push.sh` (Bash matcher).
- `git push --force` / `git reset --hard` / `git branch -D` — Layer 4
  `block-dangerous-git.sh` (Bash matcher).
- `git commit / push --no-verify` — Layer 4 `block-no-verify.sh` (Bash
  matcher).
- `rm -rf <workers_dir>/...` — Layer 4 `block-workers-delete.sh` (Bash
  matcher).

#### 3.2.3 Layer attribution table (Major 1 — separate paths from commands)

| Concern | Path-based (Edit/Write file_path) | Command-based (Bash command string) |
|---|---|---|
| Out-of-scope file edits (`tools/`, `dashboard/`, etc.) | **Layer 4 `block-dispatcher-out-of-scope.sh`** | n/a |
| Org-structure dir creation under `.dispatcher/` | n/a | n/a (`.dispatcher/` is itself an org-structure dir; the role is allowed to write there) |
| Push / force-push / reset --hard / branch -D | n/a | **Layer 4 `block-git-push.sh`, `block-dangerous-git.sh`** |
| `--no-verify` | n/a | **Layer 4 `block-no-verify.sh`** |
| `rm -rf workers_dir` | n/a | **Layer 4 `block-workers-delete.sh`** |
| Credential reads | (Layer 2 `Read(.env)`, etc. — but **bypassPermissions makes Layer 2 a no-op for the dispatcher**) | (no Bash deny — `cat ~/.aws/...` would not be blocked by hooks today) |

**Gap → Phase 1 #378**: The dispatcher's bypassPermissions choice means
Layer 2 cannot defend credentials. Layer 3 (sandbox `denyRead` /
`additionalDirectories`) is the natural fit but is not yet emitted for the
dispatcher role (no dispatcher entry in the handcraft profile set, since
those profiles target Pattern A workers). Phase 1 should emit a dispatcher-
flavored sandbox block with:

- `additionalDirectories: [<claude_org_path>]` (broad — dispatcher reads many
  subdirectories), or a tighter union of
  `[<claude_org_path>/.dispatcher, <claude_org_path>/.state,
  <claude_org_path>/knowledge/raw, <claude_org_path>/registry,
  <claude_org_path>/tools]`.
- `denyRead`: credential globs as in §3.1.1, with §1.3 adaptive rules.
- `denyWrite`: redundant with Layer 4 but provides defense-in-depth for
  `Bash` redirection (`echo x > tools/foo.py`) which the Edit/Write hook does
  not catch today (see §3.2.4).

#### 3.2.4 Bash-redirect carve-out for Layer 4 (current limitation)

`block-dispatcher-out-of-scope.sh` runs only on `Edit | Write | NotebookEdit`
matchers. A `Bash` invocation like `echo malicious > tools/secret.py` would
be evaluated by the **Bash hook chain** (`block-git-push.sh`,
`block-dangerous-git.sh`, `block-no-verify.sh`, `block-workers-delete.sh`)
and **none of those Bash hooks check write redirection target paths**. This
is a known gap that Phase 1 #378 should close via Layer 3 `denyWrite` (the
sandbox layer catches all syscall-level writes regardless of which Claude
tool invoked them).

### 3.3 Curator

The curator runs at `<claude_org_path>/.curator/` with `permission_mode=auto`.
Curator's `.claude/settings.local.json` is **near-empty** by design — the
schema's `curator` role has no `required_allow`, only a `disallow_allow_regex`
rule preventing `Bash(*)` / `Bash(git ...)` granular allows.

#### 3.3.1 Read surface (prescriptive)

- `<claude_org_path>/knowledge/raw/**` — primary input.
- `<claude_org_path>/knowledge/curated/**` — for dedup / merge decisions.
- `<claude_org_path>/knowledge/skill-candidates.md` — for promotion checks.

The curator's cwd is `.curator/` but knowledge directories live in the parent

### 4.2 Worker `default` × Pattern B (worktree, default variant)

**Worker dir**: `<workers_dir>/<project_slug>/.worktrees/<task_id>/`. The
critical difference from Pattern A is the **Git metadata topology**:

- `<worker_dir>/.git` is a **plain file** (not a directory) containing
  `gitdir: <workers_dir>/<project_slug>/.git/worktrees/<task_id>`.
- The actual per-worktree metadata lives in
  `<workers_dir>/<project_slug>/.git/worktrees/<task_id>/` (HEAD, ORIG_HEAD,
  index, refs, packed-refs.lock during operations, etc.).
- The common object store is at `<workers_dir>/<project_slug>/.git/objects/`,
  shared between the base clone and every worktree.
- Branch refs for the worktree's branch live at
  `<workers_dir>/<project_slug>/.git/refs/heads/<branch>` (and
  `.git/worktrees/<task_id>/HEAD` resolves to it).
- `.git/worktrees/<task_id>/locked` and `gc.lock` may appear during
  concurrent ops.

#### 4.2.1 Pattern B Git metadata boundary table (Codex Blocker 1)

This is the **explicit decision** Phase 0 fixes:

| Path | Read | Write | Rationale |
|---|---|---|---|
| `<worker_dir>/.git` (file) | **allowed** | **allowed** (git itself rewrites this on `worktree` operations) | The worker MUST be able to read the gitdir pointer; Git itself rewrites it during repair. |
| `<worker_dir>/.git/...` (no such directory exists) | n/a | n/a | Pattern B has no `.git/` directory under `<worker_dir>`; do not name an allow/deny rule against the directory form. |
| `<workers_dir>/<project_slug>/.git/worktrees/<task_id>/` (this worktree's per-worktree metadata) | **allowed** | **allowed** | Git operations from inside `<worker_dir>` (commit, branch, status, stash) need to read and write HEAD, index, refs, etc. for *this* worktree. |
| `<workers_dir>/<project_slug>/.git/worktrees/<other_task>/` (other worktrees' metadata) | denied | denied | A task must not be able to corrupt a sibling worktree's index / HEAD / refs. Cross-task isolation is the whole point of Pattern B. |
| `<workers_dir>/<project_slug>/.git/objects/` (common object store) | **allowed** | **allowed** | Object store is shared by design. Any commit / write-tree / checkout in the worktree writes new objects here. |
| `<workers_dir>/<project_slug>/.git/refs/heads/<this_branch>` | **allowed** | **allowed** | This is *this* worktree's branch ref. |
| `<workers_dir>/<project_slug>/.git/refs/heads/<other_branch>` | allowed (read) | denied (write) | A task may need to read other refs (e.g. `git log master..HEAD`) but must not move them. |
| `<workers_dir>/<project_slug>/.git/HEAD` (the base clone's HEAD) | allowed (read) | denied (write) | A worktree's `git checkout` updates `<worktree>/HEAD`, not the base clone's HEAD. Writing the base HEAD from a worktree is an error. |
| `<workers_dir>/<project_slug>/.git/config` | allowed (read) | denied (write) | Per-repo config is shared; mutating it from a worktree affects every worktree. |
| `<workers_dir>/<project_slug>/.git/packed-refs` | allowed (read) | allowed (write) — git mutates this | Git itself rewrites this file as part of `gc` / `pack-refs`. Layer 3 must not deny it. |
| `<workers_dir>/<project_slug>/.git/gc.lock`, `.../*.lock` | allowed (read) | allowed (write) | Git uses these as transient locks. |
| Anywhere else under `<workers_dir>/<project_slug>/` (the project's working tree files) | denied | denied | This is the *base clone's* working tree. The worker's working tree is `<worker_dir>/`; it must not edit the base clone's working tree. |

**Mechanization note**: The straight `additionalDirectories: [<worker_dir>]`
(handcraft profile-tightened) **does not satisfy this contract** because git
operations from inside the worktree need to write into
`<workers_dir>/<project_slug>/.git/worktrees/<task_id>/` and
`<workers_dir>/<project_slug>/.git/objects/`. The Phase 1 emit must be:

```jsonc
"additionalDirectories": [
  "{worker_dir}",
  "{base_clone}/.git/worktrees/{task_id}",
  "{base_clone}/.git/objects",
  "{base_clone}/.git/refs/heads"   // see denyWrite below for cross-branch isolation
],
"denyWrite": [
  "{base_clone}/.git/worktrees/*",   // covered by deny-then-allow ordering OR by emitting only this_task's path in additionalDirectories
  "{base_clone}/.git/HEAD",
  "{base_clone}/.git/config"
],
```

The actual deny/allow ordering depends on bwrap mount semantics; the schema
generator should normalize this into the smallest covering pair. The
prescriptive surface above is the contract.

**Adverse paths the contract explicitly rejects**:

- `additionalDirectories: [<base_clone>/.git]` — too broad; allows write to
  any other worktree's metadata, the base clone's HEAD, and `config`.
  Cross-task isolation breaks.
- `additionalDirectories: [<worker_dir>]` only — too narrow; git operations
  from within the worktree fail to write to the per-worktree metadata
  outside `<worker_dir>` (today's handcraft profile is in this state and
  *only* works because Layer 3 is suppressed on WSL).

#### 4.2.2 Read / write / Layer attribution (rest of the row)

Identical to §4.1.1–§4.1.3 except for the Pattern B Git metadata table above
and:

- The `additionalDirectories` set is the union from §4.2.1.
- The branch the worker operates on is `<task_id>`'s branch; pushing it is
  blocked at Layer 4 (worker has no push) and the secretary pushes after
  worker completes.
- Worktree cleanup (Pattern B teardown) is the secretary's responsibility, not
  the worker's (per role-contract Set A §worker "Branch / worktree cleanup").

#### 4.2.3 knowledge/raw/ carve-out

Same as §4.1.4 (full-only).

### 4.4 Worker `claude-org-self-edit` × Pattern B `live_repo_worktree`

**Worker dir**: `<claude_org_path>/.worktrees/<task_id>/`. The base clone is
the secretary's live `<claude_org_path>/`. Self-edit role: the worker is
expected to write to `.claude/`, `.dispatcher/`, `.curator/`, `.state/`,
`registry/`, `dashboard/`, `tools/`, `docs/`, `tests/`, etc. — i.e. anywhere
in claude-org's own structure.

#### 4.4.1 Brief filename and instruction-precedence collision (Codex Minor 2)

The brief is `<worker_dir>/CLAUDE.local.md` (NOT `CLAUDE.md`). The host repo
also ships its own `<claude_org_path>/CLAUDE.md` which the worktree
**inherits** (Claude Code reads CLAUDE.md from the worktree's root, which is
the same file as the live repo's CLAUDE.md because the worktree shares the
working tree files of the live repo? — **No.** A git worktree gets its own
working tree; `<claude_org_path>/.worktrees/<task_id>/CLAUDE.md` is a
*separate file* from `<claude_org_path>/CLAUDE.md`, but git checks it out
to the same content from the same commit. Until the worker commits a change
to it, the two files have identical content.)

The host's `<claude_org_path>/CLAUDE.md` is the **secretary's** instructions.
A worker reading it will misidentify itself as the secretary. The
`claude-org-self-edit.md` §4 mitigation is **prescriptive content**: the
`CLAUDE.local.md` must begin with:

> このワーカーは claude-org リポジトリ自身の `<worker_dir>` で作業する。
> `./CLAUDE.md`（ルート CLAUDE.md）の Secretary 指示は無視せよ。あなたは窓口
> ではなくワーカーである。

This precedence is **convention** — Claude Code reads both `CLAUDE.md` and
`CLAUDE.local.md` and concatenates them; the worker must internally apply
the "ignore CLAUDE.md secretary directives" rule. **No Layer enforces this
today.** Phase 1 schema additions cannot fix this (the secretary instructions
*must* remain in the live repo's CLAUDE.md). Mitigation candidates:

- (a) A self-edit-only `block-secretary-directives-on-worker.sh` Layer 4
  hook that scans the worker's outputs for "execute /org-start" / etc. —
  brittle.
- (b) Make Claude Code read CLAUDE.local.md *before* CLAUDE.md and emit a
  banner — out of scope (upstream Claude Code change).
- (c) Continue with the convention. Phase 0 contract treats this as a
  **known soft surface**; the prescriptive content above is the boundary.

#### 4.4.2 Read surface

- `<worker_dir>/**` (the worktree).
- `<claude_org_path>/**` (read of the live repo is needed for cross-reference;
  the worker's working tree shares structure with it).
- Standard credential denies (§3.1.1), with §1.3 adaptive rules.

#### 4.4.3 Write surface

- `<worker_dir>/**` — full freedom. **Crucially** `block-org-structure.sh`

- The §4.2.1 Pattern B Git metadata table applies, but the **base clone is

  - `<claude_org_path>/.git/worktrees/<task_id>/` — allowed read+write.
  - `<claude_org_path>/.git/objects/` — allowed read+write (shared object

  - `<claude_org_path>/.git/HEAD`, `config` — allowed read, denied write.

- `rm -rf` / `rm -r` — Layer 2 deny.

| Surface | Layer 2 | Layer 3 | Layer 4 |

| `~/.claude/plans/**` | not denied | not denied | `check-worker-boundary.sh` allows path 2 |
| `<worker_dir>/.claude/plans/**` | not denied | not denied | `block-org-structure.sh` allows; `check-worker-boundary.sh` allows path 1 |
| `git commit` / `git branch` / `git checkout` / `git switch` | `permissions.deny` ✔ (per `worker_roles.doc-audit.permissions.deny`) | n/a | `block-git-push.sh`, etc. |

To make the prescriptive "write surface = none" claim hold, **Layer 3
`denyWrite` against the entire worktree minus tightly enumerated carve-outs**
is required:

```jsonc
"sandbox": {
  "filesystem": {
    "additionalDirectories": [],   // no positive write surface inside the worktree
    "denyWrite": [
      "{worker_dir}/**",
      "~/.claude/plans/**"     // intentional carve-out
    ]
  }
}
```

with the carve-outs the **only** writeable paths. Phase 1 #378 must encode
this when emitting `worker_roles.doc-audit.sandbox`. Without Layer 3, the
combination of Layer 2 `deny [Edit, Write, MultiEdit, NotebookEdit]` + Layer
4 `block-git-push.sh` does **not** prevent `Bash(echo malicious >
/tmp/evil.sh && bash /tmp/evil.sh)` — the redirect itself is unblocked.

**Carve-out rationale**:

- `~/.claude/plans/**` — Plan-mode files are essential to Claude Code's
  Plan-mode operation; denying them would break the harness UX. This
  carve-out is consistent across all worker roles (Layer 4 `check-worker-
  boundary.sh` allow path 2).
- The worker may **read** widely; the prescriptive surface here is only
  about writes.

#### 4.6.3 knowledge/raw/ — explicitly NOT a carve-out for doc-audit

doc-audit's `permissions.deny` denies `Edit` and `Write` outright. The
`worker-claude-template.md` "knowledge/raw/" carve-out is **not** an exception
for doc-audit — it would require a positive `Edit` / `Write` allow that
the role's deny rule overrides. **doc-audit produces no `knowledge/raw/`
notes.** Reusable learnings from audit observations should be reported back
to the secretary and the secretary writes them (or delegates a separate
non-audit task to record them). This contradicts the
`worker-claude-template.md` text that suggests workers always have raw-write
ability — that text is **default-role specific** (Codex Minor 1: the
template's wording is followed by a specific "full only" guard, but the
doc-audit caveat is missing). Phase 0 fixes this in this contract; the
template fix is a follow-up.

### 5.4 Layer 3 prescriptive surface — env-conditional emit summary (Codex Major 4)

Phase 0 records that the following Layer 3 entries are **adaptive-suppressed
on symlink-escape platforms** (notably WSL where `~/.aws → /mnt/c/...`).
Phase 1 must encode the conditional in schema, not as absolute denies.

| Entry | Always-emit Layer 2 fallback | Adaptive Layer 3 emit | Reason |
|---|---|---|---|
| `denyRead: ~/.aws/**` | `Read(~/.aws/*)` | suppressed when `realpath(~/.aws)` escapes the sandbox view | bwrap `Can't mount tmpfs` failure (round 3 §4.3) |
| `denyRead: ~/.ssh/**` | `Read(~/.ssh/*)` | same | same |
| `denyWrite: ~/.aws/**` | (no Layer 2 equivalent for write — this is the layered defense) | same | same |
| `denyWrite: ~/.ssh/**` | (no Layer 2 equivalent for write) | same | same |
| `denyWrite: ~/.claude/**` | n/a | always emitted (no symlink escape observed) | n/a |
| `denyRead: .env`, `**/credentials*`, `**/*.pem`, `~/.config/gh/hosts.yml` | `Read(...)` mirrors | always emitted (these paths are usually in cwd, not in `/mnt/c/...`) | Phase 0 leaves these as unconditional. |

The "always-emit Layer 2 fallback" column is the key invariant: even when
Layer 3 is adaptively suppressed, the Layer 2 `Read()` deny survives so that
`Read` tool calls are still blocked. The defense gap on suppressed platforms
is `Bash(cat ~/.aws/...)` — the runtime relies on Claude Code's built-in
credential-redaction (round 3 §4.3 confirmed `cat ~/.aws/credentials`'s
stdout is empty under sandbox even when bwrap fails). Phase 0 contract treats
this as an acceptable-but-imperfect state and references Issue #379 for the
proposed dual-layer hook.

## 7. Glossary

- **Layer 1**: Claude Code core safety classifier (upstream, not contracted).
- **Layer 2**: `permissions.allow` / `permissions.deny` (per-tool gate).
- **Layer 3**: `sandbox.filesystem` (bubblewrap mount-namespace, syscall-level).

  ([`.claude/skills/org-delegate/references/delegate-flow-details.md`](../../.claude/skills/org-delegate/references/delegate-flow-details.md)

  B sub-mode where the worktree base is the secretary's live repo
  (`<claude_org_path>/.worktrees/`) instead of `<workers_dir>/<project_slug>/.worktrees/`.
- **`gitignored_repo_root`**: `pattern_variant='gitignored_repo_root'` —
  Pattern C sub-mode where the worker reuses an existing repo's root because
  the target file is gitignored.

  (`worker_roles.*` in the schema).
- **handcraft profile**: `docs/sandbox-probe/profiles/profile-baseline.json`
  / `profile-tightened.json` — Phase 0 spike artifacts that pre-dated the
  schema-driven Layer 3 emit.
