# Design Journal: Self-Growing Claude Code Organization

このドキュメントは、設計のやり取りの経緯と意思決定の根拠を記録したものである。
設計が上手くいかなかった場合に、何を試し、なぜその判断に至ったかを参照して仕切り直すために使う。

---

## プロジェクト概要

- **目的**: アライドアーキテクツ株式会社向けに、自己成長するClaudeCode組織を構築する
- **コンセプト**: 人間が1つのClaudeCodeと対話するだけで、裏側でワーカーが派遣・管理され、組織が学習し改善していく
- **初回設計日**: 2026-04-05
- **リポジトリの変遷**:
  - 当初は template `happy-ryo/aainc-wezterm` から派生 (WezTerm ベース)
  - 2026-04 後半に ccmux ベースの運用 repo (private) へ移行し、aainc-wezterm は引き続き template として残置

---

## 設計の経緯

### Phase 1: 初期構想 — CLAUDE.md中心設計

最初のアプローチは、CLAUDE.mdに組織の全手順を書き込む設計だった。

**構想**:
- ルートCLAUDE.md に憲法（ミッション、通信プロトコル、役割システム、成長プロトコル）
- `/roles/` に役割定義ファイル
- `/processes/` にプロセス文書
- `/knowledge/` に知見蓄積

**問題点**:
- CLAUDE.mdが肥大化し、コンテキストを圧迫する
- 全手順が常にロードされ、作業能力が落ちる
- 拡張時にCLAUDE.mdの編集が必要

→ **棄却理由**: コンテキスト消費が大きすぎる

### Phase 2: 窓口 + マネージャー + ワーカー

人間は窓口Claudeだけを使い、マネージャーClaudeが常駐して知見整理を行う設計。

**構想**:
- 窓口Claude: ユーザー対応、判断、委譲指示、結果集約（自分では実作業しない）
- マネージャーClaude: 知見整理、プロセス改善（常駐）
- ワーカーClaude: 実作業（一時起動）

**問題点**:
- CLAUDE.mdに委譲手順・知見整理手順をすべて書く必要がある
- 同じディレクトリで窓口とマネージャーが起動するため、同じCLAUDE.mdを読む問題

→ **ここでスキルの概念を導入すべきとのフィードバックを受けた**

### Phase 3: スキル中心設計（採用）

CLAUDE.mdは最小限（10行以下）にし、具体的手順をスキルに委ねる設計。

**スキルの利点**:
- プログレッシブ・ディスクロージャー: 必要なときだけ詳細手順がロードされる
- 独立進化: 各スキルが独立してバージョン管理・改善できる
- CLAUDE.mdの肥大化を防ぐ
- 拡張がスキルフォルダ追加だけで済む

**マネージャーの再定義**:
- 常駐マネージャーClaude → 専用ペインで `/loop 30m /org-curate` に変更
- マネージャーという独立した役割ではなく、org-curateスキルの定期実行として再定義

## 主要な設計判断とその根拠

### 判断1: ディレクトリ名 `aainc`

**候補**: `org`, `hq`, `headquarters`, `office`, `company`, `atelier`, `aainc`

**経緯**:
1. 最初は `org`（汎用的すぎる）
2. `hq` を提案（短くて明確）
3. 「省略形にしたい理由は？」と問われ、短さ以外の理由がなかった
4. アライドアーキテクツの企業利用が明確になり、`aainc` を提案
5. メールドメインに使われており社内で浸透済み → 採用

**教訓**: 省略形は「社内で浸透しているか」が判断基準。浸透していない省略形は避ける。

### 判断2: スキル名の `org-` プレフィックス

**理由**: ClaudeCodeのネイティブコマンド（`/resume`, `/compact`, `/clear` 等）との競合回避。

**経緯**: 最初は `delegate`, `resume`, `suspend` 等の素の名前だったが、`/resume` はネイティブコマンドと被ることが判明。`org-` プレフィックスで統一。

**教訓**: スキル名を決める前にネイティブコマンド一覧を確認する。

### 判断3: 人間が使うClaudeCodeは1つだけ

**理由**: UXの単純さ。複数のClaudeCodeを使い分けるのは人間の認知コストが高い。

**帰結**:
- 窓口Claudeが全ての委譲・集約を担当
- ワーカーは裏側で自動管理
- キュレーターは固定ペインで自動実行（人間は意識しなくて良い）

### 判断4: `.state/` を .gitignore にする

**理由**: `.state/` にはペインID等のマシン固有情報が含まれ、git管理に不適。

**トレードオフ**: 状態は共有されない。複数マシンで使う場合、各マシンの状態は独立。

**代替案として検討したもの**: なし。セッション状態の共有は不要と判断。

### 判断5: `knowledge/raw/` を .gitignore にする

**理由**: rawは整理前の一時データ。curatedに統合されれば不要。

**トレードオフ**: 整理前にマシンが壊れたらrawは失われる。

**許容理由**: rawの損失は「最近の数件の学びが失われる」程度。致命的ではない。

### 判断6: cron ではなく /loop でキュレーション

**経緯**: 最初はcronまたは `schedule` スキルを検討していた。

**変更理由**: `/loop` スキルなら専用ClaudeCodeペインで実行でき、ファイル数に応じたトリガーが自然に書ける。cronだと「ClaudeCodeを起動→コマンド実行→終了」の管理が必要になる。

### 判断7: 中断・再開メカニズム

**問題**: ClaudeCodeインスタンスは揮発的。端末を閉じると全コンテキストが失われる。

**解決策**: 三層の状態管理
1. **継続ジャーナリング**（journal.jsonl）— 全重要アクション後に追記。クラッシュリカバリ用
2. **定期スナップショット**（org-state.md）— マイルストーン後に上書き。人間可読
3. **明示的サスペンド**（org-suspend）— 最高品質の状態保存

**設計の鍵**: org-state.md が「新しいClaudeCodeインスタンスが読むだけで状況を把握できる」フォーマットであること。JSONではなくmarkdownを選んだのはこのため。

### 判断8: 自動ブリーフィング

**経緯**: 「起動時に自動で状態報告するか、/org-resume の明示呼び出しを待つか」を議論。

**結論**: 自動ブリーフィング。理由: 人間が毎回 /org-resume と打つのは面倒。CLAUDE.mdの起動時指示で自然に発動させる。

### 判断9: 窓口は司令塔、実作業はワーカーに委譲（委譲ファースト）

**経緯**: 最初は「単純な作業は自分でやり、並列化が必要なら委譲」という設計だった。

**変更**: 「窓口は司令塔であり、手を動かす実作業は原則としてワーカーに任せる」に変更。

**理由**:
- 窓口が自分で作業すると、その間ユーザーとの対話が止まる
- 委譲をデフォルトにすることで、窓口は常にユーザーの相談・判断に応じられる
- 非エンジニアが使う場合、窓口は「何でも聞ける受付」であるべき

**帰結**:
- org-delegate の description を「並列作業が必要なとき」から「実作業が発生する場合」に変更
- CLAUDE.md の原則を「単純な作業は自分でやる」から「実作業はワーカーに任せる」に変更
- getting-started.md の会話例を全面改訂

### 判断10: 非エンジニア対応（翻訳者原則 + registry メニュー化 + ダッシュボード）

**問題**: 非エンジニアは「~/work/project-a にREADMEを作成して」とは言えない。

**解決策（3層）**:
1. **registry/projects.md をメニュー化**: 通称・よくある作業例を追加。「ブログ」と言えばパスが解決される
2. **CLAUDE.md に翻訳者原則**: 技術用語を避ける、曖昧な依頼には選択肢を提示、業務言語で報告
3. **org-dashboard スキル**: 組織全体像をHTMLダッシュボードとしてブラウザで確認可能

**ダッシュボードの設計判断**: テンプレート（HTML/CSS/JS）とデータを分離。スキルはライブサーバー（`dashboard/server.py`）を起動し、サーバーが `.state/` と `registry/` を直接読んで `/api/state`（REST）と `/api/events`（SSE）でデータを配信する。テンプレートはサーバーから取得したデータをレンダリングする。デザイン変更はテンプレート側のgit commitで全環境に反映される。

### 判断11: ワーカー用 CLAUDE.md 自動生成（指示の二重化）

**問題**: ワーカーへの指示が claude-peers の send_message のみに依存していた。メッセージは揮発的で、ワーカーが長時間作業すると最初の指示を忘れる可能性がある。特に振り返り記録は作業完了時に行う必要があるが、最初のメッセージでしか指示されていなかった。

**解決策**: org-delegate がワーカー起動前にプロジェクトディレクトリに CLAUDE.md を生成する。

**二重化の設計**:
- CLAUDE.md = ベースライン（プロジェクト文脈、行動規範、振り返り指示）。起動時に確実に読まれ、セッション中ずっと有効
- instruction-template（claude-peers）= 補足（タスク固有の詳細指示）。リアルタイムだが揮発的
- 両方に完了報告・振り返り・SUSPEND対応を記載。片方が効かなくてもどちらかが機能する

**既存 CLAUDE.md への対処**: `<!-- org-worker-start -->` / `<!-- org-worker-end -->` マーカーで囲んで追記。org-suspend 時にマーカー部分を削除してクリーンに戻す。

**org-retro の役割変更**: ワーカーが CLAUDE.md の指示で自動的に技術的知見を記録するため、窓口の org-retro は「委譲プロセス自体の振り返り」（指示の出し方、タスク分解の適切さ等）に役割が変わった。

**教訓**: 揮発的な通信（MCP メッセージ）に重要な指示を一本化しない。永続的なファイル（CLAUDE.md）との二重化で堅牢性が確保される。

### 判断12: ペインレイアウトの固定化

**問題**: ペインの開き方にルールがなく、ワーカーが増えるとレイアウトが混沌とする。窓口が見えなくなると人間が状況を把握できない。

**解決策**: 固定レイアウトを定義した。

```
┌──────────────┬──────────┬──────────┐
│              │ Worker1  │ Worker4  │
│  Secretary   │ Worker2  │ Worker5  │
│  (large)     │ Worker3  │ Worker6  │
├──────────────┤          │          │
│  Curator     │  ...     │  ...     │
│  (small)     │          │          │
└──────────────┴──────────┴──────────┘
```

**設計判断**:
- 窓口を左列に大きく固定: 人間が常に窓口との対話を見られる
- キュレーターは窓口の下に小さく: 普段は見なくてよいが、動作確認はできる
- ワーカーは右側に3段ずつ列を増やす: 段数に上限なし（実運用で問題が出たら検討）
- ワーカー完了時はペインを閉じる: 画面の圧迫を防ぐ

**分割操作の要点**:
- キュレーター: 窓口を `split-bottom`（20%）
- ワーカー1人目: 窓口を `split-right`
- ワーカー2〜3人目: 右側最後のワーカーを `split-bottom`
- ワーカー4人目〜: 列が3段に達したら `split-right` で新列

### 判断13: フォアマン導入（窓口のロック回避）

**問題**: 窓口がワーカーのペイン起動・ピア待ち・指示送信を逐次処理すると、その間ユーザーとの対話がロックされる。ワーカー3人で数分のロックが発生し得る。

**解決策**: フォアマン（Foreman）を常駐インスタンスとして導入。窓口はタスク分解とCLAUDE.md生成まで行い、ペイン起動以降をフォアマンに丸投げする。

**役割分担**:
- 窓口: プロジェクト名前解決 → タスク分解 → CLAUDE.md生成 → フォアマンに依頼（ここで解放）
- フォアマン: ペイン起動 → ピア待ち → 指示送信 → 状態記録 → 窓口に完了報告
- ワーカーからの進捗/完了報告は引き続き窓口が受ける（中央集権は維持）

**設計判断**:
- フォアマンは窓口の下にキュレーターと横並びで配置（小さいペイン）
- フォアマンはワーカーの完了時ペインクローズも代行する
- org-suspend 時、フォアマンとキュレーターは最後に閉じる（ワーカーの停止処理を先に行うため）
- フォアマンを導入しても新しいスキルは不要。org-delegate の手順を窓口パートとフォアマンパートに分割しただけ

**ペインレイアウトの変更**:
```
┌──────────────┬──────────┬──────────┐
│              │ Worker1  │ Worker4  │
│  Secretary   │ Worker2  │ Worker5  │
│  (large)     │ Worker3  │ Worker6  │
├───────┬──────┤          │          │
│Foreman│Curat.│  ...     │  ...     │
└───────┴──────┴──────────┴──────────┘
```

## 未検証のリスク

### リスク1: スキルのトリガー精度
スキルのdescriptionがClaudeCodeの自動判断に使われるが、意図通りに発動するかは未検証。descriptionの文言調整が必要になる可能性がある。

### リスク2: claude-peersの信頼性
メッセージの到達保証がない（インメモリ）。ワーカーが終了していたらメッセージは届かない。org-suspendのPhase 2（スクレイプ）がフォールバックとして機能するかが鍵。

### リスク3: WezTermペインの上限
同時に開けるペイン数に実用的な上限がある。6-8ペイン以上で表示が崩れる可能性。org-delegateに同時ワーカー数の上限を設けるべきかもしれない。

### リスク4: コンテキスト消費
スキルのプログレッシブ・ディスクロージャーが期待通りに機能するか。複数スキルが同時にロードされた場合、コンテキストが圧迫される可能性。

### リスク5: 自己成長ループの発散
org-curateの改善提案が的外れだった場合、組織が悪い方向に進化する可能性。人間の承認プロセスが安全弁として機能するかが鍵。

### リスク6: CLAUDE.mdの起動時指示の実効性
「.state/org-state.md が存在すれば読め」という指示がClaudeCodeに確実に実行されるかは未検証。無視される場合、hooksで強制する等の代替策が必要。

## 仕切り直しの指針

設計が上手くいかなかった場合、以下の順で切り分ける:

### レベル1: スキルの調整（軽微な修正）
- descriptionの文言変更でトリガー精度を改善
- スキル内の手順を具体化/簡素化
- references ファイルの追加/修正

### レベル2: アーキテクチャの部分変更（中規模の修正）
- CLAUDE.mdへの記述追加（スキルだけでは不十分な場合）
- hooks の導入（起動時指示が無視される場合）
- 状態管理方式の変更（markdown → JSON、ジャーナリング方式の変更等）
- キュレーション方式の変更（/loop → cron、閾値の調整等）

### レベル3: コンセプトの見直し（大規模な方針転換）
- 窓口1つ → 人間が複数ClaudeCodeを使う設計に変更
- スキル中心 → CLAUDE.md中心に戻す（スキルが機能しない場合）
- claude-peers → ファイルベース通信に変更（MCPが不安定な場合）
- 自己成長ループを廃止し、人間が手動で知見管理する

### 切り分けの判断基準
- **問題がスキル固有** → レベル1
- **問題が複数スキルにまたがる / インフラ層** → レベル2
- **根本的な前提が崩れている** → レベル3

## 前提条件（これが崩れたら設計自体を見直す）

1. ClaudeCodeがCLAUDE.mdを起動時に読み込む → **必須**
2. `.claude/skills/` のスキルがClaudeCodeに認識される → **必須**
3. ターミナルマルチプレクサ (現在は ccmux) のコマンドでペインの開閉ができる → **必須**
4. claude-peers MCPでインスタンス間通信ができる → **必須**（代替: ファイルベース）
5. `/loop` スキルで定期実行ができる → **重要**（代替: cron + schedule）
6. ClaudeCodeが複数ペインで同時起動できる → **必須**

## 技術スタック

- **ターミナル/マルチプレクサ**: ccmux (IPC 経由でペイン制御)。当初は WezTerm + wezterm-panes-plugin を使用していた
- **エージェント通信**: claude-peers MCP（インメモリ、SQLiteバックエンドのピア管理）
- **AI**: Claude Code（Opus 4.6）
- **バージョン管理**: Git + GitHub (private)
- **OS**: Windows 11 Pro（bash shell）
