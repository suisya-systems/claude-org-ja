# コードレビュー・動作検証

## バグ修正レビューの方法論
修正後の出力が正しいかだけを見るのでは不十分。修正前と修正後の出力をdiffで比較することで、(1)修正が意図した箇所に効いているか、(2)意図しない副作用がないか、の両方を一度に検証できる。diff の影響行数を数えることで、レビューレポートに定量的な根拠を示せる。

## カラムマッピングバグの横展開チェック
カラムマッピングバグが発見された場合、同スクリプト内の関数を「ハードコードされたインデックス」と「ヘッダーから動的取得」の2パターンに分類し、前者を重点的にチェックすると効率的。動的取得の関数は構造変化に強いため優先度を下げてよい。

## データ変換スクリプトのレビュー原則
コードレビューだけでは正確性を担保できない。必ずソースデータ（Excel等）の実際の構造をプログラム的に確認する。ドキュメントやカラムヘッダーの想定だけで判断しない。

## CEPs分析レビュー観点チェックリスト
1. **数値の正確性**: スクリプト出力と手動レポートの数値を突合（CEPリンク件数、MAスコア、占有率）
2. **戦略分類の妥当性**: 「守る/攻める/捨てる」が競合構図を反映しているか（自動分類との差異を確認）
3. **施策の実行可能性**: 「誰が」「いつまでに」「どの程度のリソースで」が不明瞭でないか
4. **データ上の注意点の明示**: サンプル数が少ないブランドの統計的信頼性への言及があるか
5. **クロス集計の活用度**: 単なる件数羅列ではなく、戦略的インサイトが導かれているか

## バグ修正の再レビューチェックリスト
1. **ソースデータの構造確認**: 修正対象のカラム/行の実際の内容をプログラム的に確認
2. **修正前後のdiff比較**: 影響範囲が意図した箇所に限定されているか
3. **複数データセットでの検証**: 構造が異なるデータ（ブランド数の違い等）でも正常動作するか
4. **横展開チェック**: 同じパターンのバグが他の箇所にないか（特にハードコードされたインデックス）
5. **数値の突合**: 出力値とExcel実値の完全一致を確認

## 型ヒントの不整合
スクリプトの型ヒントが `-> dict` でも実際にはタプルを返すケースがある。型チェッカーが通っていなくても動作はするが、レビューでは指摘すべき。
