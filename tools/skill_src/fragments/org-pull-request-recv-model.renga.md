- **CI 完了 / merge 検出 / 24h タイムアウトの受信は多層 (CI-watch zero-miss, Refs #653 #658)**。renga 環境でも終端信号 (`CI_COMPLETED` / `PR_MERGED` / `PR_MERGE_WATCH_TIMEOUT` / `PR_MERGED_NO_RUN` / `PR_MERGED_HEAD_UNCONFIRMED` / `PR_WATCH_ABORTED`) は **`events` テーブルの canonical event を正本**とし、次の順で配送を受ける（peer-only の終端信号は廃止済み — canonical event が必ず先行する）:
  - **(B) ディスパッチャー relay = 見逃しゼロの主保証（一次受信経路）**: ディスパッチャーが `/loop 3m` 監視サイクルの relay scan ステップ（[`.dispatcher/references/worker-monitoring.md`](../../../.dispatcher/references/worker-monitoring.md) Step 5.25）で未配送の終端イベントを `event_deliveries` 配送台帳と突き合わせて scan し、`mcp__renga-peers__send_message` で窓口へ relay する（relay 層は transport を問わず成立する）。窓口はこの relay メッセージ（本文末尾 `[relay]` で直 push と区別可能）の到着で次のステップへ進める。pr-watch の直 push が silent noop でも events を直接読むため確実に届く。at-least-once（台帳 idempotency key で重複抑止）。
  - **(A) pr-watch からの直 push = 低遅延の補助**: renga では pr-watch が CI 確定の瞬間に `<channel source="renga-peers"> CI_COMPLETED: PR #<n> ...`（および `PR_MERGED` / `PR_MERGE_WATCH_TIMEOUT` / `PR_MERGED_NO_RUN` / `PR_MERGED_HEAD_UNCONFIRMED`）を in-band push する。届けば relay を待たず先へ進めてよいが best-effort（`RENGA_SOCKET` 未設定の plain shell / CI では silent noop）。push 失敗時は pr-watch が `notify_failed` イベントを fail-loud で記録し、それ自体も (B) relay の対象になる。
  - **(フォールバック) events テーブルの直接 poll**: relay も push も成立しない場合は従来どおり窓口が events テーブルを直接引く（下記「CI 完了検知の正路」節）。canonical event はどの層でも同一。
  - **各終端信号の分岐（受信層に依らず不変）**: `CI_COMPLETED` 受信 → ユーザーに merge 承認を仰ぐ → ユーザー承認 → `PR_MERGED` 受信で 2b-ii の post-merge cleanup へ。`PR_MERGED_NO_RUN` は merge は観測したが対応 run 行が見つからなかった失敗系（`tools/run_complete_on_merge.py` の `no_run` 終端）で post-merge cleanup には進めず人間判断で対処する。**`PR_MERGED_HEAD_UNCONFIRMED` も同様に post-merge cleanup には進めず人間確認 gate に倒す**（push と merge が pr-watch の poll 間に同時に滑り込み CI 未確認 head でマージされた終端 — loopback 不能のため fail-closed exit 9 で通知される。本文形は `PR_MERGED_HEAD_UNCONFIRMED: PR #<n> (head=<merged_short>, last CI-confirmed head=<baseline_short>)`。窓口の対処手順・人間提示文・journal kind / severity は下記「PR_MERGED_HEAD_UNCONFIRMED 受信時」節を参照）。`PR_WATCH_ABORTED` は watcher の異常終了を人間に知らせ手動 poll / 再起動を促す。
  - **`ORG_TRANSPORT=broker`（コード既定フレーム）の場合**: (A) の直 push は broker channel sidecar の `notifications/claude/channel` 注入になり、(B) relay は `mcp__org-broker__send_message` 経由になる。メッセージ本文の semantics・分岐は renga と同一。broker daemon 未起動 / channel sidecar unhealthy 時は (B) relay か events テーブル poll にフォールバックする
