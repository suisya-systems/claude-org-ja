- **pr-watch が CI 完了 / merge 検出 / 24h タイムアウトの瞬間に Secretary へ peer message を送る**（Issue #326）。`ORG_TRANSPORT=broker` を明示した場合は `CI_COMPLETED` / `PR_MERGED` / `PR_MERGE_WATCH_TIMEOUT` / `PR_MERGED_NO_RUN` は **broker channel sidecar の push 一次**で届く（各ペイン同居の channel sidecar `server:org-broker-channel` が `notifications/claude/channel` で窓口の idle セッションへ注入。runtime push-first 0.1.24+、transport-lab `docs/design/broker-native-roles.md` §9。channel source は `org-broker`）。無設定（運用既定）時は `notify_peer` が renga にフォールバックするため、下記 `ORG_TRANSPORT=renga` 枝と同じく `<channel source="renga-peers">` の in-band push で届く。窓口は events テーブルをポーリングせず、`CI_COMPLETED: PR #<n> ...`（および `PR_MERGED: PR #<n>` / `PR_MERGE_WATCH_TIMEOUT: PR #<n>` / `PR_MERGED_NO_RUN: PR #<n>`）の到着で次のステップへ進める。`CI_COMPLETED` 受信 → ユーザーに merge 承認を仰ぐ → ユーザー承認 → `PR_MERGED` 受信で 2b-ii の post-merge cleanup へ。`PR_MERGED_NO_RUN` は merge は観測したが対応 run 行が見つからなかった失敗系（`tools/run_complete_on_merge.py` の `no_run` 終端）で、post-merge cleanup には進めず人間判断で対処する。**push 失効時のフォールバックは送信に使われた transport に合わせる**: `ORG_TRANSPORT=broker` 明示時のみ窓口がターン冒頭で能動的に `mcp__org-broker__check_messages` で broker キューから本文を pull する（ナッジが出れば契機になりうるが idle を起こさないため能動 poll が正路。§9.6）。無設定（運用既定）で renga 経路に乗った通知は broker キューに載らないため `mcp__org-broker__check_messages` では拾えない — この場合は `<channel source="renga-peers">` の in-band push が一次で、取りこぼしは下記 events テーブルのポーリングで受ける。いずれの transport でも broker daemon 未起動 / `RENGA_SOCKET` 未設定の plain shell / CI では peer/channel 配送自体が成立しないため、従来どおり events テーブルのポーリングが最終フォールバックとなる
  - **`ORG_TRANSPORT=renga`（opt-in）の場合**: pr-watch は同じ瞬間に `<channel source="renga-peers"> CI_COMPLETED: PR #<n> ...`（および `PR_MERGED: PR #<n>` / `PR_MERGE_WATCH_TIMEOUT: PR #<n>` / `PR_MERGED_NO_RUN: PR #<n>`）を in-band push する。メッセージ本文の semantics・分岐は broker と同一。RENGA_SOCKET 未設定の plain shell / CI では peer-send は silent noop となり、従来どおり events テーブルのポーリングにフォールバックする
